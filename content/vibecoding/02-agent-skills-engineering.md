# AI写代码总走捷径？agent-skills 的工程化实践

最近 GitHub trending 上有个项目涨势很猛——**addyosmani/agent-skills**。

短短时间，星标已经突破 53.8k，分支也超过了 5k。作者是谷歌大佬 **Addy Osmani**，长期深耕 Chrome、开发者体验和 AI 领域。

虽然这不是谷歌官方出品，但项目里到处渗透着大厂的工程基因：测试、代码审查、性能优化、安全策略、发布流程、架构决策……这些都是 Google 一线实践沉淀下来的"隐形财富"。

我深度体验了一番，发现它最大的价值，**不是又多了一堆提示词，而是把高级工程师的思考框架，直接封装成了 AI 能调用的"技能包"**。

---

## 01 它到底是什么？

一句话：**这是给 AI 编程助手准备的工程化"外挂"**。

现在很多人用 Claude Code、Cursor、Gemini CLI 写代码，大家普遍遇到一个痛点：**AI 太爱走捷径了**。

- 需求边界还没聊清楚，AI 已经开始输出代码了
- 单元测试没补，AI 信誓旦旦说"没问题"
- 代码改动量巨大，AI 想一次性全塞给你
- 上线风险没评估，AI 就催着你合并

**agent-skills 就是来治这些毛病的。**

它把软件开发切成了一个个标准工序：从需求澄清、方案设计、增量开发、测试驱动，到代码审查、安全加固、性能调优、上线发布，全给你安排得明明白白。

AI 不再是拿到一句话就瞎写，而是按一套成熟的工程 SOP 来执行。

---

## 02 项目里藏着什么宝？

根据官方文档，它直接提供了 **7 个开箱即用的快捷命令**，覆盖了开发的完整生命周期：

| 命令 | 作用 |
|------|------|
| `/spec` | 想清楚，你要做什么 |
| `/plan` | 把大任务拆碎 |
| `/build` | 小块小块地实现 |
| `/test` | 用测试证明代码靠谱 |
| `/review` | 合码前的质量卡点 |
| `/code-simplify` | 干掉复杂，保持简单 |
| `/ship` | 准备发布上线 |

除此之外，仓库里还有 **23 个细分技能**。其中 22 个覆盖了从编码到上线的每一个环节，剩下的 1 个是"总管家"，专门判断当前该调哪个技能。

> 看完这目录，我最大的感触是：**这不是"提示词合集"，这是一本 AI 可执行的"工程规范手册"**。

---

## 03 实操体验：像有个技术大牛在旁边盯着

我最喜欢这个项目的一点是，**不用一次性全塞给 AI，按场景按需加载就行**。

### 场景1：做新功能先跑需求澄清

```
/spec
```

它不会立刻甩代码，而是反问我：

- 目标是什么？
- 边界在哪？
- 接口怎么定？
- 测试怎么验？
- 风险有哪些？

这种体验很像旁边坐了一位极其负责的架构师——既不允许你糊弄需求，也不允许 AI 糊弄实现。

### 场景2：拆解任务

```
/plan
```

以前 AI 的毛病是一次性改十几个文件，出错根本没法查。现在它把任务拆成小步，**每一步都能验证**，容错率高太多了。

### 场景3：落地编码

```
/build
```

它遵循**"极薄切片"原则**：写一小块，跑通测试，再继续下一块。做大项目重构时，这个思路简直是救命稻草。

---

## 04 这四个技能最让我惊喜

### 测试驱动开发

很多 AI 生成的测试不是"假断言"就是空壳。这个技能**强制 AI 围绕业务行为写测试**，让代码真正被验证。这对后端逻辑和复杂函数特别友好。

### 代码审查与质量

这个技能相当于把 **Google Staff Engineer 的审查清单**塞给了 AI。它不只看"风格"，而是死磕正确性、可维护性、测试覆盖率和风险边界。

### 安全加固

只要代码涉及登录、权限、用户输入、DB 操作、文件上传或第三方接口，这个技能就是"护身符"。**AI 经常忽略的安全盲区，它都能扫一遍**。

### 代码简化

这个专门用来"治"AI 写出的那些绕来绕去的代码。它的哲学很简单：**别为了炫技写复杂，能删就删，能直白就直白**。

## 05 支持哪些工具？很广

这也是它爆火的原因——**不挑食**。

| 工具 | 使用方式 |
|------|----------|
| Claude Code | 支持插件安装 |
| Cursor | 把 SKILL.md 放进 rules 目录 |
| Gemini CLI | 直接安装技能 |
| 其他 | Windsurf、OpenCode、GitHub Copilot、Kiro、Codex……只要支持系统提示词或规则文件，都能用 |

这玩意本质上就是**一堆 Markdown 文件**。你的 AI 编程工具只要能读指令，就能用上这套方法论。

## 06 适合谁？别盲目冲

如果你只是写几十行脚本，这项目可能感知不强。

但如果你已经开始用 AI 做真实项目——改业务代码、补单元测试、重构、查线上 Bug、写复杂接口、整理 PR——**那这个项目非常值得花时间研究**。

最匹配的三类人：

1. **AI 编程重度用户**：你会发现 AI 不再是"莽夫"，干活有条理多了
2. **技术负责人/架构师**：你可以把这套 Skill 作为团队的 AI 开发规范，统一质量底线
3. **想进阶的工程师**：哪怕不用 AI，光看这些技能文档，都能学到高级工程师的工作习惯

---

## 07 几点忠告

项目虽强，但不是银弹。

- Skill 能约束 AI 的工作流程，但**不能保证 AI 永远不出错**
- 复杂的业务逻辑、敏感的生产数据、关键的安全权限、最终的上线决策，**还是得人亲自把关**

另外，**技能不是越多越好**。一次性加载全部规则，AI 可能会变慢、变啰嗦，甚至抓不住重点。

> 我的建议：先从 `/spec`、`/plan`、`/build`、`/test`、`/review` 这几个核心环节开始。用顺手了，再逐步加上安全、性能、发布类的专项技能。

## 08 最后说点真心话

agent-skills 最打动我的，是它把 AI 编程往前推了一大步：

> **从"只会写代码"进化到了"会按工程流程做事"。**

传统提示词解决的是"这一次该怎么回答"，而这套 Skills 解决的是"**以后遇到这类任务该怎么干**"。

它拿下 53.8k Star，我毫不意外。

AI 编程的下一阶段，拼的或许不再是底层模型算力，而是**谁能把顶尖工程师的经验，沉淀成 AI 可复用的标准化流程**。

从这个角度看，agent-skills 已经站在了浪潮的前端。
